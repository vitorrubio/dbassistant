# DB Assistant

## Overview
DB Assistant is a layered .NET 10 solution that exposes a REST API for translating natural language questions into safe read-only MySQL queries for connected business databases. The application business is database connectivity plus natural-language querying, so it uses direct SQL access instead of an ORM `DbContext`. The current implementation uses a hybrid schema-context strategy: it searches an indexed schema knowledge base first and then falls back to live metadata from `INFORMATION_SCHEMA`, so newly created tables remain discoverable even when they are not yet indexed in the RAG layer.

The container image is published to GitHub Container Registry as `ghcr.io/vitorrubio/dbassistant`. The expected public package page is `https://github.com/users/vitorrubio/packages/container/package/dbassistant`.

## Local Development
Clone the repository and create your local environment file from `.env_copy-me`. Fill the placeholders in `.env` with your MySQL and OpenAI credentials before running the API. MySQL configuration uses individual environment variables (`MYSQL_HOST`, `MYSQL_PORT`, `MYSQL_DATABASE`, `MYSQL_USERNAME`, `MYSQL_PASSWORD`), and the application composes the connection string internally.

```bash
git clone <repository-url>
cd dbassistant
cp .env_copy-me .env
docker compose up --build api
dotnet restore DBAssistant.sln
dotnet build DBAssistant.sln
dotnet test DBAssistant.sln
dotnet run --project tools/DBAssistant.KnowledgeGenerator/DBAssistant.KnowledgeGenerator.csproj
./scripts/docs/build.sh
./scripts/docs/serve.sh
dotnet run --project src/Api/DBAssistant.Api/DBAssistant.Api.csproj
```

The API exposes Swagger at `/swagger` and the first endpoint at `POST /api/assistant/query`.

The current production Swagger URL is `https://dbassistant.icywave-4d743178.brazilsouth.azurecontainerapps.io/swagger/index.html`. This URL is exposed by the Azure Container App named `dbassistant` after the `ci.yml` deployment job completes.

To pull and run the published image:

```bash
docker pull ghcr.io/vitorrubio/dbassistant:latest
docker run --rm -p 8080:8080 --env-file .env ghcr.io/vitorrubio/dbassistant:latest
```

The response JSON currently contains:

```json
{
  "columns": ["ColumnA", "ColumnB"],
  "rows": [
    {
      "ColumnA": "value",
      "ColumnB": 10
    }
  ],
  "resultsAsText": "Short Markdown answer or table generated from the SQL result."
}
```

When `showDetails=true`, the response also includes `sql`, `explanation`, `schemaContextSource`, and `executed`. When `showDetails` is omitted or `false`, those fields are hidden. When `executeSql` is omitted, it defaults to `true`.

`OPENAI_MODEL` defaults to `gpt-5.4`. The local RAG bootstrap reads from `knowledge/runtime/schema-documents.json` by default; if no relevant schema knowledge is found, or if runtime generation fails, the application uses `INFORMATION_SCHEMA` directly.

Additional optional environment variables:
- `OPENAI_EMBEDDING_MODEL`: embedding model used for vector retrieval over schema documents.
- `CACHE_SQL_PLAN_MINUTES`: in-memory TTL for repeated SQL plans.
- `CACHE_SCHEMA_SEARCH_MINUTES`: in-memory TTL for repeated schema retrieval results.
- `SCHEMA_KNOWLEDGE_DIRECTORY`: runtime directory where the generator writes schema artifacts.
- `SCHEMA_KNOWLEDGE_EMBEDDING_INPUT_FILE_PATH`: JSONL file produced by the generator with one embedding input per document.
- `SCHEMA_KNOWLEDGE_EMBEDDINGS_FILE_PATH`: local persisted embeddings file for schema documents.

The assistant endpoints are protected by a caller API key. Send the configured value in the header defined by `API_KEY_HEADER_NAME` when calling `POST /api/assistant/query`. The default public header is `x-api-key`.

The RAG artifacts are generated by `tools/DBAssistant.KnowledgeGenerator`. It connects to the configured MySQL server, reads schema metadata and foreign keys from `INFORMATION_SCHEMA`, and writes the runtime retrieval artifacts before the API starts:
- `schema-documents.json`: enriched retrieval documents for database overview, tables, relationships, join paths, lookup tables, fact tables, and views.
- `schema-embedding-input.jsonl`: one JSONL record per document with the embedding-optimized text.
- `schema-embeddings.json`: persisted embedding vectors materialized at startup so the first production request does not need to embed the full corpus on demand.

These artifacts are generated at runtime into the configured knowledge directory and are intentionally not committed, not copied into the image as pre-generated assets, and not published as deployment artifacts. The container startup runs `DBAssistant.KnowledgeGenerator` before the API, precomputes the schema embeddings when OpenAI configuration is available, and continues with the existing `INFORMATION_SCHEMA` fallback if generation fails.

The automatic project documentation is built with DocFX from the repository sources, XML code comments, and the generated Swagger 2.0 description exported from the API assembly. The published static site is intended to be served by GitHub Pages for this public repository.

## Changelog
### 2026-03-24
- Materialized `schema-embeddings.json` during runtime startup so semantic schema retrieval no longer builds the full embedding cache inside the first user request after deploy or restart.
- Evolved the schema knowledge generator into a runtime RAG corpus builder that emits enriched schema documents and embedding-input JSONL files, switched container startup to generate artifacts before the API boots, and kept the existing `INFORMATION_SCHEMA` fallback when runtime generation is unavailable.
- Added a DocFX-based documentation pipeline with XML code comments, generated REST API reference, local docs scripts, and a GitHub Pages workflow for the public project site.
- Upgraded the assessment readiness story with function-calling style SQL planning, typed OpenAI transport contracts, vector-based schema retrieval with embeddings, in-memory caching, richer Swagger metadata and examples, evaluator-focused compliance documentation, and a real-LLM evaluation harness for the official prompts.
- Consolidated the architecture documentation under `docs/architecture`, translated the detailed documents to English, and prepared Mermaid diagrams for direct visualization from Markdown.
- Hid `schemaContextSource` and `executed` from assistant responses whenever `showDetails` is omitted or `false`, keeping metadata visible only in explicit detail mode.
- Added a new Portuguese architecture documentation set under `docs/arquitetura`, including component/class diagrams, a technical architecture document, and technical decisions with CI/CD, secrets, and cost rationale.
- Added header-based API key protection for the assistant endpoints and wired Azure Container App deployment to load the API key value from a managed secret.
- Published the Docker image to GitHub Container Registry and updated CI to restore, build, run only categorized unit tests, build the container, and prepare Azure Container App deployment from the immutable SHA tag.
- Moved local debug HTTPS settings to `launchSettings.json`, added Markdown result previews to assistant responses, and hardened OpenAI Responses parsing for structured output.

### 2026-03-23
- Bootstrapped the solution structure, environment templates, Docker assets, CI workflow, Swagger API, and the first read-only SQL assistant flow.
- Added hybrid schema context assembly with RAG-first lookup and `INFORMATION_SCHEMA` fallback, and fixed the default OpenAI model to `gpt-5.4`.
- Simplified Docker Compose to run only the API container because the target MySQL database is externally hosted.
- Migrated the solution, CI workflow, and Docker images to .NET 10.
- Removed the unused `FinTechXDbContext` and EF Core dependencies, keeping data access focused on direct SQL queries through `MySqlConnector`.
- Added a generated, Git-tracked Northwind RAG artifact and a knowledge-generator tool that rebuilds the schema index from the source database.
